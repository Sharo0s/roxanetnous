## What worked / Successes

### 1. Audit MCP BDD prod en amont du cadrage

L'utilisation du MCP Supabase pour auditer l'état réel de la BDD prod **avant** de rédiger le tech-spec 5.A.1 a évité un dérapage majeur. La divergence doc archi vs prod (4 erreurs structurelles : enum, helpers RLS, tables profils, scope colonnes) aurait causé une migration cassée si la story 5.A.2 avait suivi naïvement le doc archi. Pattern reconductible : **toujours valider via MCP avant cadrage BDD critique**.

### 2. Checkpoint review `bmad-checkpoint-preview` post-cadrage

Le passage par `bmad-checkpoint-preview` après les 4 steps de cadrage a permis de découvrir et corriger des sous-estimations en amont de l'exécution (15 colonnes vs 4 annoncées, 23 policies vs 7+). Si on avait découvert ça pendant 5.A.2, c'eût été une révision de tech-spec en cours d'exécution, friction garantie. **Pattern à institutionnaliser** : checkpoint review systématique avant exécution d'un mini-epic à risque BDD.

### 3. Stratégie cutover atomique (F12) bien dimensionnée

Décision D1 du tech-spec 5.A.1 : rejet expand-contract au profit du cutover atomique. La volumétrie prod (822 users, 9 tables vides) a confirmé que l'expand-contract aurait été du sur-engineering. Cutover effectif < 5 sec sans incident. **Leçon** : adapter la stratégie de migration à la volumétrie réelle, pas aux best practices génériques.

### 4. Re-évaluation des "blocages" hérités

Le cadrage initial Epic 5 marquait 5.B comme "post-5.A merged" (verrouillage par convention). Re-évaluation en cours d'exécution : 5.B.1 et 5.B.3 ne touchaient pas la BDD ni les fichiers re-typés par 5.A.3, donc démarrables en parallèle. De même F-Epic5-0 a levé le blocage "fenêtre go-live" après évaluation du risque réel. **Pattern** : les contraintes de séquencage du cadrage initial sont des hypothèses, pas des vérités. Re-évaluer en exécution avec les faits.

### 5. Garde-fous CI durcis transversalement

3 nouveaux scripts CI livrés Epic 5 :
- `check-as-any-global.mjs` : extension story 4.6 admin à tout le code TS.
- `check-oracle-paywall.mjs` : garde-fou anti-régression sécurité messagerie.
- `vercel-env-push.mjs` : automation push env vars (mode dry-run + apply).

Tous intégrés `vercel.json` buildCommand. **La surface de garde-fous CI passe de 4 à 6** entre Epic 4 et Epic 5. Pattern : chaque story qui introduit une convention métier doit aussi livrer un garde-fou CI qui la verrouille.

### 6. Snapshot pré-cutover comme artefact de rollback

Le snapshot `snapshot-5-a-2-pre-cutover.md` (enum + 23 policies DDL + counts users + body helpers) a été un investissement à coût marginal qui aurait débloqué un rollback en < 5 min en cas d'incident migration. Bien que non utilisé (cutover réussi), c'est l'assurance qui rend possible le risque calculé. **Pattern à généraliser** : tout cutover BDD prod doit avoir son snapshot DDL persisté en amont.

### 7. Instrumentation Sentry préexistante = 5.E.1 simplifié

Story 5.E.1 audit initial (52 captureException déjà taggés `severity` + `flow` uniformément depuis Epic 4) a réduit la story à de la **configuration externe** (alert rules + push notifs OS), pas du code. **Leçon** : le code applicatif d'Epic 4 a "investi" pour Epic 5. Convention de tagging précoce paye plus tard.

## What didn't work / Imparfaits

### 1. Ping-pong décision F-Epic5-E1 (Slack → Sentry mobile → Email)

3 révisions en 1 journée pour le canal oncall :
1. **Matin** : Slack recommandé sans demander si Sylvain l'utilise. Erreur d'investigation utilisateur.
2. **Après-midi tôt** : Sentry mobile app recommandée. **Erreur factuelle** : ce produit n'existe pas (vérification web search a posteriori). J'aurais dû vérifier avant de recommander.
3. **Après-midi final** : Email + push iPhone Mail. Solution simple, déjà 80% en place, qu'il aurait fallu proposer en premier.

**Coût** : 3 commits + 3 mises à jour DECISIONS.md/doc archi/epic-5.md/sprint-status. Friction utilisateur.

**Leçon** : avant de recommander un produit / plateforme, vérifier (a) qu'il existe, (b) que l'utilisateur l'utilise déjà, (c) qu'il est le moins disruptif pour le cas d'usage. 2 questions en amont auraient évité 3 révisions.

### 2. Cadrage initial F-Epic5-0 trop conservateur

Le cadrage initial marquait les 3 audits 7j Epic 4 (AI-4.1, AI-4.2, AI-4.3) comme "bloquants" pour Epic 5. Re-évaluation en exécution : seul AI-4.3 (toggle admin) impacte le go-live, les 2 autres sont des observations passives. **Coût** : 1 demi-journée perdue à m'opposer à l'exécution avant que Sylvain me ramène à la raison ("je trouve pas pourquoi tu attends").

**Leçon** : les pré-requis hérités d'un epic précédent ne sont pas automatiquement applicables au suivant. Auditer chaque pré-requis individuellement pour confirmer la causalité réelle.

### 3. Find-replace global accidentel (URL routes / nom de fichiers components)

Le `perl -i -pe` sur `/accompagnante/` a remplacé toutes les références URL **mais aussi des imports `@/components/accompagnante/...`** qui n'avaient pas encore été renommés. Conséquence : 3 erreurs `Cannot find module` qu'il a fallu débuger après tsc. **Pattern défaillant** : find-replace de masse sans whitelist explicite. **Correction immédiate** : `git mv components/accompagnante components/accompagnant` après le coup.

**Leçon** : pour les renommages massifs touchant fichiers + paths d'imports, faire les `git mv` AVANT les find-replace, dans l'ordre topologique. Ou utiliser un outil dédié (codemod / jscodeshift) qui gère les imports.

### 4. Erreurs TS exposées par retrait `as any` (5.C.1)

La résorption des 14 `as any` hors admin a exposé 20 erreurs TS cachées (nullable narrowing + mismatch type `experience: number` vs `string`). Toutes corrigées, mais ce dérapage de scope aurait dû être anticipé. La mémoire `project_blocage_ci_brownfield` documentait déjà ce risque ("48 erreurs hors-admin sur 17 fichiers" exposées story 4.6). J'aurais dû relire cette mémoire avant 5.C.1.

**Leçon** : avant de toucher au typage TS strict, relire les mémoires projet pertinentes pour éviter de redécouvrir un dérapage déjà documenté.

### 5. Dette pré-existante `experience: number` vs `string` non résorbée

Story 5.C.1 a documenté un bug pré-existant dans `lib/matching.ts` : `AccompagnanteProfile.experience` typé `string` mais `auxProfile.experience` reçu `number` depuis BDD. Conséquence : score localisation matching probablement faussé depuis longtemps. **Décision** : laisser tel quel + cast `String()` localisé + commentaire explicite, correction reportée Epic 6.

**Leçon** : c'est acceptable de différer une dette quand sa résorption est hors scope, mais il faut s'assurer que la dette est trackée (AI Epic 6).

### 6. `.swc/` accidentellement commit

Le `git add -A` du commit 7228947 a inclus `.swc/.gitignore` + un fichier wasmer cache (artefacts build local Next.js). Corrigé immédiatement commit `53ef706`. **Leçon** : préférer `git add <files>` explicite à `git add -A` pour les commits importants. Ou s'assurer que `.gitignore` couvre toutes les sorties de build en amont.
