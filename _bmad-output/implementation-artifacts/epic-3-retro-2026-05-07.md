## What worked / Successes

- **Centralisation `lib/departements.ts` + tag-based cache `unstable_cache` 30 s** : réutilisé sans duplication par 5 call-sites parallèles (`lib/matching.ts`, `app/recherche/`, `components/landing/DepartementsOuverts.tsx`, server action onboarding, server action waitlist). Invalidation immédiate au toggle admin via `revalidateTag(DEPARTEMENTS_CACHE_TAG)`. Aucun round-trip BDD chaud sur les 5 pages. **Pattern à dupliquer** pour toute table « lue souvent, écrite rarement » à l'avenir (catégories d'aide, abonnements types, paramètres globaux).

- **Pattern audit matrice métier reproductible** : story 3.6 explicitait une matrice 12 actions × 3 rôles (visiteur / connecté non abonné / abonné) → grep des 12 call-sites → triage `DECISION_NEEDED / DEFER / PATCH`. Story 3.1 a appliqué le même pattern à la matrice géographique. Résultat : **zéro incohérence métier non documentée** au merge, rapport audit `_bmad-output/implementation-artifacts/3-6-audit-paywall.md` reproductible par n'importe quel reviewer. **Pattern à appliquer** à toute future feature impliquant une matrice de droits ou de comportements.

- **Idempotence niveau BDD plutôt qu'applicative** : story 3.4 `INSERT ... ON CONFLICT (lower(email), code_departement) DO NOTHING` capture violation `23505` → retourne `alreadyRegistered: true`. Story 3.5 `UPDATE waitlist SET notified_at = now() WHERE id = ? AND notified_at IS NULL` (compare-and-swap atomique). Aucune race possible même si admin + cron déclenchent simultanément. **Anti-pattern évité explicitement** : pas de check-before-insert applicatif (race window + leak PII).

- **Defense en profondeur server action + UI disabled + heuristique client** : pattern systématisé sur 3.2 (onboarding + lien waitlist), 3.4 (form waitlist), 3.6 (paywall messaging). Server action = source de vérité (sécurité). UI disabled / redirection / heuristique d'erreur = couche UX. Aucun bypass devtools détecté par Edge Case Hunter sur les 8 stories.

- **Code reviews adversariales 3-layers obligatoires** : 8 stories sur 8 ont déclenché Blind Hunter + Edge Case Hunter + Acceptance Auditor. Triage structuré (patches / defer / dismiss) tracé dans chaque PR. Aucun finding bloquant resté ouvert au merge. **La convention forgée Epic 2 (story 2.3 review multi-agent 2026-05-04) est désormais le standard de facto** sur tout l'épic.

- **Pattern fail-loud + audit trail** : story 3.8 raffine le pattern issu Epic 2 patch H5. Si `ADMIN_NOTIFICATIONS_EMAIL` manque → `console.warn` explicite + insert `admin_actions_log` action_type=`parrainage_admin_alert_lost` + libellé fr visible dans `/admin/historique`. Plus jamais de signal anti-fraude perdu silencieusement. **Convention maintenant claire** : tout fail (Resend timeout, env manquante, validation BDD échoue) → audit trail + console.warn + cron retry capture. Pas de perte.

- **0 % de dérive d'effort par story** sur les 8 stories Epic 3. Pas d'extension de scope en cours de revue (la discipline « ajouter une story dédiée plutôt qu'élargir » du Lot C 2.7.6 a été reproduite implicitement sur Epic 3).

## What didn't work / Imparfaits

- **Bug latent schema BDD `admin_actions_log.target_id`** : `UUID NOT NULL` mais `app/admin/departements/actions.ts` insérait un code TEXT depuis le toggle admin. Aucun toggle dpt n'avait jamais été exécuté en prod → bug dormant depuis Epic 2. Découvert en pré-cadrage story 3.5. **Mitigation** : migration ajoute `target_id_text TEXT NULL`, mutex applicatif (commentaire SQL + docstring) car CHECK BDD non envisageable rétroactivement (historique). **Leçon** : les chemins de code non exécutés en prod ne sont pas testés tant que la migration n'a pas été activée. Un test manuel « toggle un dpt en staging » Epic 2 aurait révélé le bug. **À reproduire** : pour toute nouvelle table avec colonnes contraintes, ajouter au moins un test manuel exécutable en staging dans la story qui crée la migration.

- **Bug latent `logNotification` schema drift transversal** : `status='error'` rejeté par CHECK BDD + `user_id NOT NULL` empêche logging pour visiteurs anonymes. Touche **15+ flows email** (cron parrainage, waitlist confirmation, ouverture dpt, admin notif, reset password, etc.). Workaround Epic 3 : skip log si `userId` undefined + use `status='failed'` au lieu de `'error'`. **Pas une vraie solution** : la dette persiste, le bug se déclenche au premier visiteur anonyme qui devrait être tracé. **Décision** : fix transversal `user_id NULLABLE` + `type ENUM` au lieu de CHECK fragile **bloquant pour go-live Epic 4**.

- **Email synchrone bloquant dans server action** : story 3.4 (`submitWaitlist` await `sendWaitlistConfirmationEmail`), story 3.5 (`toggleRegion` notifie séquentiellement avec `BATCH_LIMIT=200`). Pire cas 3.5 : 13 dpt × 50 inscrits × 300 ms Resend = ~3 min → timeout server action 60 s. Cap + cron retry capturent l'overflow mais c'est un workaround. **Vraie solution** : queue durable (Vercel Workflow DevKit, Vercel KV polling, ou job runner dédié). **Bloquant pour go-live Epic 4** car l'ouverture du premier dpt déclenchera le burst.

- **Tests e2e absents sur 8/8 stories** : convention projet MVP héritée. AC test manuel documenté dans chaque PR. Couverture zéro régression flux financiers (Stripe webhook), parrainage anti-fraude, paywall, RGPD cascade, matching. **Risque croissant** à mesure que le code grossit. **Bloquant pour go-live Epic 4** au moins sur Stripe webhook + paywall (flux argent).

- **IP spoofing trivial via `x-forwarded-for` non sanitizé** sur rate-limits 3.4 / 3.6 (pattern hérité parrainage). Vercel sanitize en prod, mais hors-prod = trou. **Pattern à durcir** : `request.ip` Vercel API ou `x-vercel-forwarded-for` (header trusted Vercel-only).

- **Frontière feature vs prerequisite poreuse** : stories 3.7 (audit cookies RGPD) et 3.8 (robustesse notif admin) ne sont pas des features mais des dettes pré-go-live. Inclure ou non dans l'épic « lancement Bretagne » a été un arbitrage explicite. **Leçon** : les stories non-fonctionnelles bloquantes pour go-live doivent être listées comme AC du PRD (FR ou NFR), pas cachées en « best practice ». Sinon, dev et PO découvrent le désaccord en sprint planning.

- **Action items rétro Epic 2 non traités Epic 3** : 3 sur 4 deferred-work issus de la review 2.3 du 2026-05-04 ne sont **pas** soldés à la clôture Epic 3 :
  - Rate-limit `validateCode` parrainage : ⏳ pattern `try_consume_rate_limit` posé story 3.4 mais pas appliqué à `validateCode`.
  - Alerting Sentry sur `console.error` transverse : ❌ non traité.
  - Types `as any` pages admin : ❌ non traité (candidat I reporté Epic 4).
  - Validation `escapeHtml` test XSS templates email : ❌ non traité.

  **Lecture** : Epic 3 était scope « lancement Bretagne », pas « hardening ». Acceptable mais doit être explicité au prochain cadrage : les engagements rétro qui ne sont pas re-priorisés en début d'épic suivant **glissent automatiquement**. **Solution** : intégrer une revue des deferred-work au début de chaque sprint planning.

## Décisions prises pendant l'épic (à acter dans DECISIONS.md)

- **F1 — Géographie : Whitelist `departements_ouverts` 5 dpt Bretagne + FK CASCADE sur `waitlist_departements`** :
  - Whitelist stockée table `departements_ouverts` (22, 29, 35, 44, 56).
  - Toute nouvelle feature affectant la géographie respecte la whitelist (filtrage lecture, blocage inscription, etc.).
  - FK CASCADE sur `waitlist_departements` → purge automatique si dpt supprimé (acceptable car data anonyme purpose-specific).
  - Refresh cache `DEPARTEMENTS_CACHE_TAG` immédiat au toggle admin (pas de délai 30 s).
  - Extension géographique future : ajouter régions à `departements_ouverts` + maintien FK sans down-time migration.

- **F2 — Soft Paywall messaging : paywall sur création conversation, pas lecture historique** :
  - `sendMessage` : paywall obligatoire sauf conversation avec admin.
  - `getOrCreateConversation*` : paywall sur **création**, pas lecture (utilisateur qui a perdu abonnement peut consulter historique).
  - UI defense-en-profondeur : button disabled + modal `/abonnement?from=contact` si non-abonné (UX, pas sécurité).
  - Server action est source de vérité.

- **F3 — `admin_actions_log` schema dual-target avec mutex applicatif** :
  - `target_id UUID NULLABLE` : pour actions sur utilisateurs/subscriptions.
  - `target_id_text TEXT NULLABLE` : pour actions sur codes (dpt, région).
  - Applicatif mutex : exactement un des deux est set. **Pas de CHECK BDD** pour préserver historique.
  - Documentation commentaire SQL + docstring.

- **F4 — Notifications admin : fail-loud + audit trail** :
  - Perte de signal anti-fraude inacceptable.
  - `sendAdminParrainageFlag()` si email manque → insert `admin_actions_log` action_type=`parrainage_admin_alert_lost`.
  - Libellé fr dans `/admin/historique` (mapping `actionLabels` synchronisé).
  - `console.warn` (config absente) vs `console.error` (vraie panne Resend/DB).
  - Build-time garde-fou `scripts/check-required-env.mjs` warning si `ADMIN_NOTIFICATIONS_EMAIL` manquant en `VERCEL_ENV=production` (non-bloquant Epic 3, à durcir Epic 4).

- **F5 — Waitlist idempotence : UNIQUE INDEX case-insensitive + compare-and-swap notify** :
  - `INSERT waitlist` capture `23505` unique violation → return `alreadyRegistered: true`. UNIQUE INDEX `(lower(email), code_departement)`.
  - `UPDATE waitlist SET notified_at WHERE id = ? AND notified_at IS NULL` compare-and-swap atomique.
  - Une personne ne reçoit jamais deux emails pour la même ouverture, même en cas de toggle bug ou race condition.

## Go-live Bretagne — décision Project Lead

**Décision 2026-05-07 : go-live Bretagne pilote conditionné à livraison Epic 4 partiel (4 axes bloquants).**

Le go-live n'est **pas** déclenché à la clôture Epic 3 contrairement à la recommandation initiale Amelia (J+3 stabilisation). Les 4 axes Epic 4 suivants sont déclarés bloquants par le Project Lead :

1. **AI-3.1** Fix schema `logNotification` transversal.
2. **AI-3.2** Tests métier critiques (a minima Stripe webhook + paywall).
3. **AI-3.3** Email queue durable.
4. **AI-3.4** Rate-limit `validateCode` parrainage + alerting Sentry transverse.

**Rationale** : ouvrir le service en prod réelle avec 4 dettes critiques connues (perte signal email, zéro tests automatisés flux argent, risque timeout 60 s, erreurs prod invisibles) serait prématuré. Le coût de déboguer un incident en prod sans visibilité Sentry et sans tests dépasserait le coût de livrer Epic 4 partiel d'abord.

**Conséquence** : Epic 4 cadrage à lancer immédiatement après cette rétro. Epic 4 doit livrer les 4 AI bloquants avant d'autoriser le toggle admin du premier département.
