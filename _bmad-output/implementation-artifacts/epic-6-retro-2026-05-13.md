## What worked / Successes

### 1. Méthodologies institutionnalisées Epic 5 (AI-5.M.1, AI-5.M.2) appliquées

Le tech-spec 6.A.1 a démarré par un **audit MCP BDD prod re-run** (AI-5.M.2) qui a confirmé l'inventaire et révélé deux corrections critiques **avant** l'écriture du SQL :
- `ALTER TYPE DROP VALUE` n'est PAS implémenté en Postgres 17 (test BEGIN/ROLLBACK = `0A000`). L'epic-6.md prétendait l'inverse — vérifié au lieu d'être cru.
- Bug latent dans le trigger `handle_new_user` qui n'aurait pas été détecté sans relire la fonction (route les signups accompagnant vers la table accompagnes_profiles).

**Sans ces deux découvertes**, le premier `apply_migration` aurait crashé (DROP VALUE non supporté) et le second crash silencieusement (trigger insérant dans une table renommée). Pattern confirmé : **les méthodologies AI-5.M.1/M.2 paient cher leur coût marginal**.

### 2. Bugs latents découverts par effet de bord du renommage

Le simple alignement terminologique a révélé **10 bugs latents prod** qui n'auraient probablement jamais été trouvés autrement :

1. Trigger `handle_new_user` route mal les signups accompagnant (`handle_new_user.IF v_role_text IN ('accompagnante', ...)` vs `auth.ts:20` qui envoie `'accompagnant'`).
2. Redirect Stripe post-success vers `/accompagnante` 404 depuis Epic 5 (route renommée mais subscription.ts:45 jamais aligné).
3. URL signup `/register?role=accompagnante` rejetée par auth.ts qui attend `'accompagnant'`.
4. URL filtre admin `?type=accompagnante` rejetée par handler (TYPE_VALUES strict depuis story 5.B.3).
5. `ROLE_LABELS.accompagnante` lookup → `undefined` dans 5 fichiers admin (clé absente vs `user.role='accompagnant'`).
6. Tests intégration paywall cassés silencieusement (colonne `conversations.accompagnante_id` n'existe plus depuis Epic 5).
7. RPC orpheline `get_or_create_conversation` jamais appelée (0 caller TS depuis 3 mois).
8. Code mort `lib/ocr.ts` (278 lignes, caller supprimé commit `0585867` 2026-02-18 sans cleanup).
9. Matching nullable safety + commentaire faux `'experience est un number en BDD'` (faux — `text`).
10. Divergence seed départements prod (UTF-8 accentué) vs migration originale (non accentué) — `supabase db reset` cassait dev/CI.

**Leçon** : un epic "cosmétique" est en réalité un audit transverse révélateur. Pattern reconductible : les renommages massifs servent aussi de smoke test sémantique.

### 3. Échec contrôlé du 1er apply_migration → rollback automatique transactionnel

Le 1er `apply_migration` 6.A.2 a échoué (`0A000: cannot alter type of a column used in a policy definition`). **Transaction rollback automatique** par Postgres → BDD prod **strictement intacte**. Verified : enum encore avec `accompagnante`, tables encore au féminin, 0 row touchée.

L'audit immédiat post-échec a identifié 3 policies RLS bloquantes (`parrainages_codes_admin_full`, `parrainages_admin_full`, `users_update_own`) que le snapshot D8 du tech-spec n'avait pas capturées (cherchait `is_accompagn` dans qual/with_check, pas `users.role` direct). Migration révisée avec DROP/CREATE de ces 3 policies + recréation post-ALTER COLUMN TYPE → 2e apply success.

**Leçon** : la transactionalité DDL de Postgres + un guard upfront (refus si `users.role = 'accompagnante'` existe) sont l'assurance qui rend le "directement en prod sans branche preview" acceptable. Pattern Epic 5 5.A.2 → Epic 6 6.A.2 confirmé sur 2 cutover prod sans incident.

### 4. Décision d'embarquer la dette dans la même migration

Plutôt que de saucissonner par story (renommage tables → 6.A.2, renommage contraintes → 6.A.x, fix trigger → 6.B.4, etc.), la décision F-Epic6-A1 a embarqué tout dans **une seule transaction atomique** :
- 3 RENAME tables
- 14 RENAME contraintes (F-Epic6-A1-b)
- 11 + 3 DROP/CREATE policies RLS (cosmétique + bloquantes)
- recreate-enum (F-Epic6-A1-d)
- fix trigger `handle_new_user` (F-Epic6-A1-e)
- DROP RPC orpheline
- table vide `accompagne_accompagnantes` aussi embarquée (F-Epic6-A1-a)

Coût cumulé : ~5 secondes downtime. **Si on avait saucissonné** : 6 fenêtres cutover successives, 6 snapshots, 6 plans rollback, 6 fenêtres de validation. Pattern à institutionnaliser : **les migrations DDL pures (metadata) peuvent être groupées agressivement** sans pénalité runtime.

### 5. Re-évaluation du scope 6.D avec courage de reporter

Le mini-epic 6.D promettait "extension de la suite Playwright story 4.4". L'audit 2026-05-13 a révélé que **la suite Playwright fonctionnelle n'existe pas** (4.4 a livré Vitest intégration, Playwright est uniquement utilisé pour axe-core a11y). Le scope réel : créer l'infra E2E (~1j) + 3 scénarios applicatifs (~0.5j chacun) = 2-3j-dev.

Plutôt que de bricoler une infra E2E à la va-vite dans une soirée Epic 6, **reporté Epic 7+** avec scope clarifié dans DECISIONS.md F-Epic6-D et plan de réactivation. **Leçon** : reconnaître qu'un cadrage est faux et reporter coûte moins cher que livrer du sous-engineering qu'il faudra refaire.

### 6. Décisions confirmées par audit factuel

3 décisions ont été **validées par audit** plutôt que par intuition :
- **6.B.3 (suppression backward-compat)** : audit Sentry 30j + BDD `notifications_ouverture` → 0 trafic prod sur `?role=accompagnante`. Suppression safe.
- **6.B.2 (suppression OCR)** : audit git history → caller supprimé commit `0585867` (2026-02-18, 3 mois sans cleanup). Code mort confirmé, 278 lignes supprimées.
- **6.C.4 (rejet centralisation helpers email)** : audit factuel 20 helpers / 1243 lignes / 19 try-catch / 39 logNotification. Pattern uniforme, churn de refactor disproportionné. Rejet confirmé avec 3 conditions de réactivation Epic 7+ documentées.

**Leçon** : les décisions "garder" sont aussi importantes que les décisions "changer", et méritent d'être actées dans DECISIONS.md avec le contexte audit.

### 7. Migration corrective UTF-8 idempotente

Story 6.C.3 a livré une migration `UPDATE ... WHERE ... IS DISTINCT FROM` qui aligne dev/CI sur la prod déjà UTF-8. **Test idempotence sur prod via BEGIN/ROLLBACK** : 0 row affectée (déjà aligné). Migration applicable plusieurs fois sans effet de bord. Pattern reconductible : **toute migration de re-synchronisation doit être idempotente par construction**.

14 commits Epic 6, chacun avec une description de 20-50 lignes documentant :
- contexte audit

## What didn't work / Imparfaits

### 1. Affirmation fausse sur `DROP VALUE` Postgres 17 dans le tech-spec initial

Le **premier draft du tech-spec 6.A.1** affirmait avec confiance "`ALTER TYPE ... DROP VALUE` disponible Postgres 17+". L'audit a heureusement vérifié par test BEGIN/ROLLBACK en prod : `ERROR: 0A000: dropping an enum value is not implemented`. Tech-spec corrigé immédiatement, décisions D4 et D8 réécrites.

**Coût** : 0 (vérifié avant apply). **Leçon** : ne jamais affirmer une capacité Postgres sans test. Même les rumeurs de release notes ne suffisent pas. Vérification empirique = obligation.

### 2. Bug trigger `handle_new_user` non capté par l'audit D8 du tech-spec

Le tech-spec 6.A.1 D8 (snapshot pré-cutover) prévoyait un snapshot du body de `handle_new_user` mais **ne mentionnait pas explicitement** que le trigger contenait du code obsolète (référence à `'accompagnante'` + insert dans table renommée). Le bug n'a été découvert qu'en lisant le body du trigger par curiosité — pas par le cadrage.

**Coût** : retroactif (décision F-Epic6-A1-e ajoutée a posteriori, tech-spec révisé). **Leçon** : lors de l'audit pré-cadrage d'une migration BDD, **lire systématiquement le body des fonctions/triggers stockés** qui réfèrent les tables impactées. Pas juste les schémas.

### 3. Audit `pg_policies` du tech-spec D8 incomplet (3 policies bloquantes ratées)

L'audit dépendances du tech-spec 6.A.1 cherchait les policies référençant **les helpers `is_accompagnant`/`is_accompagne`**, pas celles référençant **directement `users.role`**. Résultat : 3 policies (`parrainages_codes_admin_full`, `parrainages_admin_full`, `users_update_own`) ratées, 1er apply_migration échoué.

Heureusement, échec transactionnel sans dégât. Mais **15 minutes perdues** à réviser le SQL + commit `bf0ceb6` qui aurait pu être intégré au commit initial `22e8fb5`.

**Leçon** : pour le pattern recreate-enum (ALTER COLUMN TYPE), l'audit doit chercher **toutes** les références `users.role` dans `pg_policies.qual/with_check`, pas juste les helpers. Mettre à jour la méthodologie AI-5.M.2 avec cette précision.

### 4. Estimation budget 6.A initial trop optimiste

Tech-spec 6.A.1 estimait initialement 2.75 j-dev pour 6.A. Révisé à 3.25 j-dev après ajout du fix trigger + snapshot étendu. **Réalité exécution** : ~3h en soirée (cadence atypique Sylvain comme Epic 5).

Le budget annoncé n'a pas matché la réalité, mais ce n'est pas un problème en pratique car la cadence Sylvain est exceptionnelle. **Leçon** : continuer à annoncer des budgets j-dev pour le scope théorique, sans extrapoler la cadence atypique observée Epic 5/6.

### 5. Scope tech-spec 6.D faussement annoncé "extension"

Le cadrage Epic 6 décrivait 6.D comme "Etendre la suite Playwright story 4.4 avec les scenarios reportes". Hypothèse fausse : 4.4 a livré Vitest intégration, pas Playwright fonctionnel. Aurait dû être vérifié au cadrage.

**Coût** : 15 minutes d'audit au lancement 6.D pour découvrir le pot aux roses, puis report Epic 7+. Pas de code écrit en pure perte (l'audit est utile pour Epic 7+).

**Leçon** : lors du cadrage d'un mini-epic "extension", vérifier explicitement **quelle infra existe**. Ne pas supposer que les noms (`Playwright`) recoupent la même chose entre stories (4.4 a11y vs 6.D E2E).

### 6. Variables internes TS au féminin non renommées (cosmétique non-bloquant)

Décision pragmatique acceptée : laisser ~84 variables/paramètres locaux au féminin (`accompagnantesCount`, `accompagnanteProfileId`, `AccompagnanteRow`, `repartition.accompagnantes`, etc.) malgré la cohérence terminologique. Renommer aurait touché 5-10 fichiers supplémentaires par variable pour 0 impact runtime.

**Coût** : dette cosmétique restante (documentée dans le commit `89e14ea`). **Leçon** : le scope "cohérence éditoriale" a une limite naturelle au-delà de laquelle le churn devient disproportionné. Tracker explicitement ce qu'on laisse au lieu de prétendre que tout est aligné.

### 7. Migration BDD `admin_actions_log` target_type appliquée hors fenêtre formelle

La migration UPDATE 6.A.4 (24 rows `admin_actions_log.target_type='accompagnante'` → `'accompagnant'`) a été appliquée **avant** la fin de la story 6.A.4, sans snapshot rollback formel comme 6.A.2. Action triviale (UPDATE idempotent), 0 risque, mais hors méthodologie AI-5.M.2 stricte.

**Leçon** : la méthodologie snapshot pré-cutover ne s'applique pas qu'aux DDL — un UPDATE 24 rows mériterait quand même un snapshot count + sample de rows avant. Affiner AI-5.M.2 : snapshot obligatoire pour `apply_migration` DDL **ou** DML > 0 rows.
