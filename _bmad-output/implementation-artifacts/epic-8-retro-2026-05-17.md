## What worked / Succès saillants

### S1 — Audit MCP BDD en story dédiée avant code (8.A.0)

L'épic démarre par un audit MCP lecture seule de 4 tables (`parrainages`, `parrainages_codes`, `users`, `subscriptions`). 13 requêtes SQL documentées, **3 décalages spec-vs-réalité rectifiés** :
- `user_role` enum 3 valeurs (`accompagnant`, `accompagne`, `admin`) — pas de `'visiteur'` mentionné dans la spec.
- `subscription_status` enum 4 valeurs avec `cancelled` **double L** — pas `'canceled'` ni `'unpaid'` mentionnés dans la spec.
- Pas d'index sur `users.role` (déferré post-Epic 8 si volumétrie l'exige).

**Verdict GO sans migration** acté dans `F-Epic8-A0`. Pattern AI-5.M.2 / AI-6.M.1-3 institutionnalisé : tout nouveau rôle parrain (futur Epic) requiert audit MCP identique avant écriture code.

### S2 — FLAG-A identifié dès l'audit et propagé en cascade (8.A.2 → 8.A.3)

L'audit 8.A.0 a flaggué que `confirmParrainageOnSuccess:789` interroge `accompagnants_profiles` inconditionnellement. **Sans correction, tout parrain accompagné aurait été silencieusement bloqué à `'marraine_no_longer_validated'`**. 8.A.2 corrige côté server action (branching `if (role === 'accompagnant') {...} else if (role === 'accompagne') {...}`), 8.A.3 propage le même pattern au cron `confirm-parrainages`. Les deux stories citent explicitement FLAG-A dans leurs Dev Notes — chaîne de transmission impeccable.

### S3 — Discipline « story de cleanup en aval » pour le wording neutre (8.C.3)

Plutôt que de renommer `sendParrainageBienvenueMarraine` dans 8.A.1 (où la fonction est utilisée pour la 1ère fois côté accompagné), la décision est de **créer directement la fonction neutre `sendParrainageBienvenueAccompagne`** et de défèrer le rename de l'historique à 8.C.3. Résultat : 8.A.1 n'a aucun risque de casser les callers accompagnants existants ; 8.C.3 introduit un **alias deprecated 1-release** pour la transition douce. Pattern reconductible.

### S4 — Branches Sentry discriminantes pour observabilité role-aware

Chaque check role-aware (`validateCode`, `confirmParrainageOnSuccess`, cron) émet désormais un signal Sentry avec `extra.roleParrain` + `extra.subStatus` (ou `extra.roleFilleul` pour les guards). 6 patches en code review 8.A.2 ont rendu cette observabilité systématique. Si demain une régression apparaît, on saura **immédiatement** quel rôle est concerné. Signaux ajoutés : `marraine-unexpected-role`, `marraine-unexpected-role-at-confirm`, `invalid-filleul-role`, `marraine-sub-inactive`, `marraine-ineligible-at-payment`, `genese-accompagne-failed`, `genese-accompagne-email-failed`, `cron-marraine-unexpected-role`, `cron-marraine-role-read-failed`, `cron-coupon-failed`.

### S5 — 3 niveaux de tests complémentaires (pas redondants)

- **Unit** (87/87) : logique branching role-aware sur mocks Supabase chainables, rapide (~1s).
- **Intégration** (6 SC, BDD réelle locale via GHA) : invariants BDD réels (enum `cancelled` double L, RLS, FK CASCADE, RPC atomique `parrainage_claim_recompense`).
- **E2E Playwright** (4 SC) : parcours utilisateur réel + assertions BDD post-paiement.

Le SC2 d'intégration (`accompagnant → accompagnant` non-régression Epic 2) verrouille l'historique par construction. Tout breaking change futur fera échouer SC2 instantanément en CI.

### S6 — Décision F-Epic8-A3 explicite sur la divergence cadrage-vs-réalité FLAG-E

L'epic-8.md mentionnait « `recompense_perdue` + reset compteur si sub inactive au palier ». Le code Epic 2 héritage `continue`-and-retry **ne reset pas** le compteur dans ce cas. La story 8.A.3 conserve la sémantique Epic 2 (cohérence Epic 2) **et formalise explicitement la divergence dans DECISIONS.md** pour éviter qu'un futur dev croit qu'il « manque » une feature. Pattern « acter les décisions garder » est aussi important que « acter les décisions changer ».

### S7 — Code review BMad systématique 3 couches sur chaque story

Blind Hunter + Edge Case Hunter + Acceptance Auditor sur chaque story Epic 8. ~24 patches appliqués au total, ~30 defers tracés explicitement. Pattern reconductible : **les findings dismissed sont également documentés** avec rationale.

### S8 — 8.D.1 E2E livré avec 2 runs GHA verts consécutifs après debug rapide

4 runs initiaux ont échoué (sélecteur CityAutocomplete sans `name`, puis dropdown autocomplete intercepte le click). 2 fixes ciblés (commits `0438400` + `85e5af3`) ont mené aux 2 runs verts consécutifs requis par AC7. Boucle de feedback courte (~15 min par cycle) grâce à l'infra Playwright Epic 7.C déjà en place.

`8-a-4-tests-integration-parrainage-symetrique 2.md` et `8-d-1-...golden-path 2.md` existent à côté des originaux. Source probable : conflit iCloud Drive ou rename macOS Finder. Pollution mineure mais à nettoyer (AI-Epic8-M2).

Cible spec dans 8.A.4 AC4. Pas exécutable localement (Sylvain ne lance pas Docker). Le GHA workflow `integration-tests.yml` ne publie pas le `coverage-summary.json` comme artefact. **Defer formalisé** mais le chiffre 85% reste théorique aujourd'hui. À solder Epic 9 (AI-Epic9-B2).

### I3 — Validation manuelle navigateur reportée systématiquement (6 stories)

Chaque story UI reporte une « validation manuelle staging Sylvain post-merge » parce qu'aucune session accompagné réelle avec abonnement+code n'existe en local. Liste : T7 8.A.1, T5 8.B.1/8.B.2, T10 8.C.1, T14 8.C.3, T7 8.D.1. Pattern documenté mais **dette systémique cumulée**. À résoudre par seed staging Epic 9 (AI-Epic9-A3).

### I4 — `epic-8: in-progress` non basculé sur `done` dans `sprint-status.yaml`

Bien que toutes les 11 stories soient `done` à la fin de la journée, la clé `epic-8` reste `in-progress` au démarrage de la rétro. Pattern à clarifier dans la convention BMad (AI-Epic8-M1).

### I5 — F1 review 8.A.1 (status hardcodé `'active'`) découvert tard

Le webhook `checkout.session.completed` passait `status: 'active'` en dur dans `triggerAccompagneCodeGenesisIfEligible` alors que `subscriptionResponse.status` était disponible. Découvert en code review post-livraison T1-T6. Le check role-aware a fonctionné par hasard (les sessions checkout passent quasi toutes en `active`), mais le code était **subtilement faux** pour le cas `trialing` immédiat.

### I6 — Mocks Supabase chainables `fromMock` non discriminants par table

Defer hérité de 8.A.1, reconduit dans 8.A.2 et 8.A.3 sans correction. Les tests unit reposent sur un dispatcher séquentiel qui ignore le nom de table. Si demain quelqu'un refactore l'ordre des requêtes, les tests passeront avec une mauvaise table. **Dette qui grandit story par story** — à solder Epic 9 (AI-Epic9-B1).

### I7 — 30+ entrées dans `deferred-work.md` après Epic 8

L'épic livre énormément, mais reporte aussi énormément. Concentration de defers ciblant « Epic 9 » : alias 1-release, rename `sendParrainageFilleulConfirmation`, types TS `FilleuleStatut`, refonte mocks Supabase, RPC `parrainage_decrement_compteur` atomique, perf optimisation (early-exit genesis, fusion lookup `users`), couverture AC4, validation manuelle UI. **Si Epic 9 ne cadre pas un mini-épic « hardening Epic 8 »**, ces defers vont s'éroder. Solution validée : AI-Epic9-A1 cadrage mini-épic dédié.

### I8 — Pas de rétrospective Epic 7 (sprint-status: `optional`)

Pallié par l'addendum « Epic 7 light » de cette rétro (ci-dessous).

### I9 — `lib/emails.ts` a `'use server'` mais importé depuis modules plain (defer 8.A.1 F6)

Pré-existant, dismissed parce que « le build Next.js valide ». Anti-pattern qui pourrait casser avec une future version Next.js App Router. À surveiller.
