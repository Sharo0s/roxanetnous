## What worked / Successes

### 1. Absorption complète des AI Epic 3 (11/11)

L'Epic 4 a converti chaque AI-3.x en story dédiée (mapping 1-pour-1) et livré la totalité des 9 actionables, plus le cadrage assumé des 2 conditionnels. **Aucun AI Epic 3 n'a glissé silencieusement** — la mémoire `project_epic_3_retro.md` qui flaggait le risque de glissement (3 sur 4 deferred-work Epic 2 non traités Epic 3) a été correctement traitée.

### 2. Cadence 2 jours pour 9 stories d'hardening

Inédit pour le projet. Tenue par : pas de feature UI complexe (durcissement transverse), patterns réutilisés (helpers purs, idempotence BDD), commits chirurgicaux, code reviews adversariales batches mais pas en cascade.

### 3. Pattern MCP Supabase comme source autoritaire

Story 4.6 (génération `types/supabase.ts` ~1700 lignes via MCP) et story 4.7 (récupération 28 migrations brownfield via `supabase_migrations.schema_migrations`) ont validé MCP comme source autorisée pour les tâches structurelles — supérieur à `pg_dump` (perd historique) ou staging dédié (coût). Pattern reconductible Epic 5+.

### 4. Découverte critique du blocage CI brownfield (story 4.7)

Le premier run GHA workflow #25502322720 a révélé que **le repo ne contenait que 25/52 migrations** : Epic 1 livré rétroactivement avant adoption BMad formelle, sans migration init. Cette découverte aurait été un blocage majeur si elle s'était manifestée plus tard. Sylvain a repriorisé 4.7 bloquant go-live ordre 1 le 2026-05-09 (cf. memory `project_blocage_ci_brownfield.md`), et la migration brownfield + seeds ont ensuite débloqué les tests intégration 4.4 (14/14 verts en 7,47 s CI).

### 5. Garde-fous CI durcis intégrés `vercel.json:buildCommand`

Pipeline buildCommand chaîné bout-en-bout au merge :
```
check:env && lint:a11y-check && check:ip-spoofing && check:as-any-admin && SKIP_E2E_TESTS=true && next build
```
Trois nouveaux scripts de garde-fou créés ou durcis Epic 4 :
- `check:env` (story 4.8) : exit 1 prod si REQUIRED manquante, silence preview pour OPTIONAL.
- `check:ip-spoofing` (story 4.5) : grep audit interdit lectures directes `x-forwarded-for`/`x-real-ip` hors `lib/get-client-ip.ts`.
- `check:as-any-admin` (story 4.6) : AST checker interdit `as any`/`as Record<string,any>`/`<any>` dans `app/admin/`.

### 6. Defense in depth pour notifications email

Le cumul stories 4.1 + 4.2 + 4.3 a livré une chaîne défensive complète :
- **4.1** : `Sentry.captureException` + tags métier sur 8 chemins critiques + PII stérilisée.
- **4.2** : schema `notifications_log` étendu (user_id NULLABLE, status 7 valeurs, ENUM type rejeté au profit TEXT évolutif).
- **4.3** : Vercel Workflow DevKit avec retry × 3 + backoff exponentiel + fallback synchrone si queue down + cron `notify-waitlist-retry` 30j safety net.

Aucun email perdu en cas de défaillance Resend OU queue OU schema. Couches indépendantes.

### 7. Tests Vitest 4 + GitHub Actions Supabase CLI

Story 4.4 a posé un nouveau pattern d'intégration backend (Vitest 4 ESM native + GHA + Supabase CLI) qui a permis de tester la signature webhook Stripe et le paywall server action sans Playwright UI overhead. 5 tests Stripe (signature, parrainage anti-fraude, payment_failed, subscription_deleted) + 5 tests paywall (non-connecté, sans abonnement, actif, expire mid-conversation, admin bypass FR11quater). Helper `stripe-webhook-helper.ts` réutilisable.

### 8. Variante locale SCP (story 4.6) sur conflit étendu

48 erreurs TS hors-admin sur 17 fichiers ont été découvertes pendant 4.6. Plutôt que tout traiter (refactor explosif), cast localisé `as unknown as SupabaseClient<Database>` sur 7 pages admin uniquement (variante locale SCP D5 élargie) + déférer le reste à story 4.6.b ultérieure. Décision pragmatique avec Sylvain.

## What didn't work / Imparfaits

### 1. Volume de patches review en forte hausse vs Epic 3

44 patches Epic 4 vs 14 patches Epic 3. Pic stories 4.3 (13 patches) et 4.4 (13 patches) reflètent :
- Courbe d'apprentissage Sentry SDK v10 / Vercel Workflow DevKit (technologies nouvelles).
- Mocking Stripe/Resend non-trivial (signatures webhook, événements duplicate, fallback synchrone).
- Hors scope élargissement fréquent (4.6 `as any` partout, 4.8 vars d'env audit, 4.9 refonte escapeHtml).

**Pattern à surveiller Epic 5** : si nouveau pattern technique introduit, prévoir spike court avant story formelle pour réduire volume patches review.

### 2. Découverte tardive du blocage CI brownfield

Le manque de migration init Epic 1 aurait dû être détecté lors du cadrage Epic 4 (revue prérequis CI), pas au premier run GHA story 4.4. Coût : **5 itérations 4.7** pour stabiliser (auth.users trigger handle_new_user, contrainte XOR T10, drift versioning 27 versions). Heureusement repriorisation rapide (1 jour) sans bloquer go-live final.

**Leçon** : pour tout futur chantier CI / migration / infra, **first run end-to-end avant lock du périmètre**. Memory `project_blocage_ci_brownfield.md` à conserver comme reference.

### 3. Variables d'environnement Sentry / RATE_LIMIT à configurer manuellement post-merge

Story 4.1 pose 5 vars Sentry + 1 RATE_LIMIT_HASH_SALT, story 4.8 formalise REQUIRED vs OPTIONAL_ON_PREVIEW, mais **build prod initialement ERROR** post-livraison story 4.8 (vars manquantes scope Production) → configuration manuelle 2026-05-08 puis redeploy `dpl_6iReah6atXaaZigGa21W3sS6twSs`. Pattern récurrent : le check enforce à juste titre, mais les setup-actions Vercel ne sont pas automatisées.

**Action Epic 5+** : checklist post-merge documentée avec liste exhaustive des vars à pousser sur Vercel + scope (Production/Preview/Dev), idéalement scriptée via `vercel env add`.

### 4. Variante locale SCP D5 élargie sur 4.6 = dette différée

Le cast localisé `as unknown as SupabaseClient<Database>` résorbe les `as any` admin **mais déplace la dette** : les factories Supabase restent non-typées génériquement. Story 4.6.b (candidat Epic 5) à cadrer pour traiter les 17 fichiers hors-admin et idéalement remplacer le cast par typage natif des factories.

### 5. Cron retry `notify-waitlist-retry` conservé 30j safety-net (story 4.3)

Décision pragmatique mais introduit dette : le cron coexiste avec la queue durable, double mécanisme. Suppression prévue story 4.3.b après audit 30 j zéro signal `signal:queue-cron-fallback-active` Sentry. À tracker Epic 5+.

### 6. BATCH_LIMIT=200 conservé sur send-waitlist-opening (story 4.3)

Conservatisme : le reviewer a suggéré la suppression (queue durable rend BATCH_LIMIT obsolète), mais Sylvain a préféré garder le filet. Pas un bug, mais un peu de complexité pas strictement nécessaire. À reconsidérer une fois la queue stable en prod.

### 7. Test endpoint `/api/test-sentry` temporaire restant story 4.1

Endpoint créé pour valider l'instrumentation Sentry au merge. À nettoyer en hotfix court Epic 5+ (TODO loggé deferred-work).

### 8. Couverture metric Vitest non-bloquante (story 4.4)

AC11 spec voulait un seuil minimum de couverture, descopé au profit de tests fonctionnels ciblés (5 + 5). Acceptable MVP, mais pas de garantie qu'une régression silencieuse sur un chemin non-couvert ne passe pas.

### 9. Fixtures inline 4.4 vs seeds globaux 4.7 = double pattern

Story 4.4 utilise des fixtures BDD inline avec cleanup `afterEach`, story 4.7 livre des seeds SQL globaux pour Epic 5+. Coexistence acceptée (D2 story 4.7) mais à clarifier dans les conventions tests : quand fixtures inline vs seeds globaux ? Convention Epic 5+ à documenter.

DECISIONS.md a déjà été enrichi de F6 à F11 pendant l'épic (cf. commits + last_updated sprint-status.yaml). À vérifier post-rétro que les décisions ci-dessous sont bien tracées :

- **F6** — Sentry SDK Vercel-compatible v10+, replays/profiling désactivés (volume/cost), instrumentation flow-centric tags + PII stérilisée.

- **F10** — IP cliente : `ipAddress(@vercel/functions)` lit `x-real-ip` Vercel Proxy sanitisé. Helpers `getClientIp` (null) et `getClientIpOrUnknown` (string fallback). Lecture directe XFF interdite (garde-fou `check:ip-spoofing`).
- **F11** — Types Supabase : génération MCP `types/supabase.ts`, factories Supabase typées `<Database>` côté admin uniquement (variante locale SCP D5 élargie). Garde-fou `check:as-any-admin` AST scope `app/admin/`. Hors-admin reporté story 4.6.b.
