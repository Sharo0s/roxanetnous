## 2. Constats par zone

### 2.1 Layout racine (`app/layout.tsx`)

- ✓ `lang="fr"` correctement défini
- ✓ Polices avec `display: 'swap'` (pas de FOIT)
- ✗ Aucun **skip-link** vers `<main>` (WCAG 2.4.1 Bypass Blocks)
- ✗ Aucun `<main>` enveloppant les enfants au layout root (chaque page le déclare ou non, incohérent)
- ✗ Pas de balise `<title>` dynamique annonçant les changements de page (mitigé par metadata Next, mais à vérifier sur navigation client-side)

**Impact** : utilisateur clavier doit retabber tout le header à chaque page.

### 2.2 Carte SVG hero (`components/landing/hero-carte.tsx`)

- ✓ `aria-hidden="true"` sur le SVG → ignoré par lecteurs d'écran (choix défendable car décoratif)
- ✗ **Aucune alternative textuelle** dans la zone (« Notre service couvre les principales villes de Bretagne : Rennes, Brest, Lorient, Vannes, etc. »)
- ✗ Animation infinie sur les `circle` (pulse `repeatCount="indefinite"`) sans respect de `prefers-reduced-motion`
- ✗ Pas de version statique alternative
- Note : si la carte porte du sens (zone de couverture du service), `aria-hidden` cache une info importante. Si purement esthétique, OK mais ajouter un texte adjacent.

**Impact** : un utilisateur non-voyant arrive sur la home page sans comprendre que le service est limité à la Bretagne. Pour basse vision sensible aux animations, les pulses peuvent être inconfortables.

### 2.3 Composant `Input` partagé (`components/ui/input.tsx`)

**Composant central** : utilisé partout dans les formulaires. Ses défauts se propagent à tout le site.

- ✗ `<label>` **sans `htmlFor`** → label/input non associés (RGAA 11.1, WCAG 1.3.1, 3.3.2)
- ✗ Erreur affichée en `<p>` non lié au champ (pas d'`id` + `aria-describedby`)
- ✗ Pas de `aria-invalid` quand erreur présente
- ✗ Astérisque `*` rouge pour « obligatoire » sans annonce textuelle (WCAG 1.3.1) — il faut un `aria-label="obligatoire"` ou texte caché
- ✗ Couleur de focus : `focus:ring-[#FFB06E]` (orange pêche) sur fond blanc → contraste **2.0:1**, échec WCAG 2.4.7 (focus visible) qui exige **3:1** minimum
- ⚠ `focus:outline-none` sans alternative robuste pour navigateurs sans support `:focus-visible`

**Impact** : aucun utilisateur de lecteur d'écran ne peut comprendre quel champ il remplit. Aucun utilisateur basse vision ne voit où il est focus.

### 2.4 Composant `Button` partagé (`components/ui/button.tsx`)

- ✓ `focus-visible:ring-2 focus-visible:ring-offset-2`
- ✗ **Pas de couleur de ring définie** dans les classes → utilise la couleur par défaut Tailwind (souvent blue-500), incohérent et parfois invisible
- ✗ Pas de `min-h-[44px]` sur taille `default` (h-10 = 40px) → cible tactile WCAG 2.5.5 Niveau AAA (mais bonne pratique)
- ✗ Pas de gestion `aria-busy` ou texte alternatif quand loading

### 2.5 Onboarding auxiliaire (`components/accompagnante/onboarding-client.tsx`)

- ✓ `<main>` présent
- ✗ Progress bar = simples divs sans `role="progressbar" aria-valuenow={step+1} aria-valuemax={4}` → invisible aux lecteurs d'écran
- ✗ « Étape X sur 4 » dans `<span>` non relié au progress
- ✗ Bloc d'erreur sans `role="alert"` ni `aria-live="assertive"` → utilisateur ne sait pas qu'une erreur vient d'apparaître
- ✗ Changement d'étape ne déplace pas le focus → utilisateur clavier doit retabber depuis le bas pour atteindre le nouveau formulaire en haut
- ✗ Bouton « Suivant » disabled sans annonce des champs manquants

**Impact** : parcours d'inscription auxiliaire **non finalisable** par un utilisateur de lecteur d'écran sans assistance.

### 2.6 Header authentifié (`components/layout/accompagne-header.tsx`)

- ✓ `aria-label="Menu"` sur le bouton burger
- ✗ Pas de `aria-expanded={menuOpen}` — lecteur d'écran ne sait pas si le menu est déployé
- ✗ Pas de `aria-controls="mobile-menu"`
- ✗ Liens du header pas dans un `<nav>` (le bloc desktop est un simple `<div>`)
- ✗ `text-gray-600` sur blanc : ratio 4.7:1 — limite AA, pas confortable

### 2.7 Cookie banner (`components/cookie-banner.tsx`)

- ✗ Pas de `role="region"` ni `aria-label="Bandeau cookies"` sur le conteneur
- ✗ Apparaît sans déplacer le focus ni annoncer (acceptable pour cookies essentiels, mais à éclaircir avec CNIL)
- ✗ Bouton « Compris » sans `:focus-visible` style explicite

### 2.8 Footer (`components/footer.tsx`)

- ✓ `<footer>` + `<nav>` corrects
- ✓ Contrastes excellents (texte noir sur `bg-accent` ~14:1)
- ⚠ `<nav>` sans `aria-label` (« Liens légaux » recommandé pour différencier de la navigation principale)

### 2.9 Messagerie (`components/messages/`)

- ✗ **Zéro attribut ARIA** dans tout le module (`grep` sans résultat)
- ✗ Pas de `role="log"` ou `aria-live="polite"` sur la liste de messages
- ✗ Pas d'annonce des nouveaux messages reçus
- ✗ Pas de `aria-label` sur la liste des conversations

**Impact** : messagerie **inutilisable** par un lecteur d'écran. Critique car cœur du produit.

### 2.10 Carte Leaflet (`components/ui/map-radius*.tsx`)

Non auditée en détail mais Leaflet a des problèmes a11y connus et nécessite configuration explicite (rôles, alternatives clavier). À traiter dans la phase de remédiation.

## 3. Contrastes — analyse de la palette

| Combinaison | Ratio | Niveau AA (4.5:1 texte / 3:1 UI) |
|---|---|---|
| `text-black` sur `bg-accent` (#000 / #F4C8A3) | ~14:1 | ✓ |
| `text-black` sur `bg-kraft` (#000 / #d3a387) | ~9.5:1 | ✓ |
| `text-gray-500` sur blanc | 4.83:1 | ✓ limite |
| `text-gray-500` sur `bg-accent` | **3.4:1** | ✗ texte normal |
| `text-gray-600` sur blanc | 7.6:1 | ✓ |
| `border-gray-300` (#D1D5DB) sur blanc | **1.6:1** | ✗ UI < 3:1 |
| `focus:ring-[#FFB06E]` sur blanc | **2.0:1** | ✗ < 3:1 |
| `focus:ring-[#FFB06E]` sur `bg-accent` | ~1.1:1 | ✗ totalement invisible |
| `text-red-500` (#EF4444) erreur sur blanc | **4.0:1** | ✗ limite |

**Action** : durcir les bordures de champs (`border-gray-400` minimum), changer la couleur de focus pour un orange foncé ou un bleu/noir système, passer `text-red-500` à `text-red-700` pour les erreurs.

## 4. Animations & motion

`globals.css` définit :
- `animate-marquee` (25s linéaire infini)
- `animate-marquee-reverse` (idem)
- `animate-blink` (1s infini)
- `animate-fade-in` (0.8s, ponctuel — OK)
- Animations SVG `repeatCount="indefinite"` sur la carte hero

**Aucune media query `@media (prefers-reduced-motion: reduce)`** → violation WCAG 2.3.3 et 2.2.2 (Pause, Stop, Hide).

**Action** : ajouter dans `globals.css` :

```css
@media (prefers-reduced-motion: reduce) {
  .animate-marquee,
  .animate-marquee-reverse,
  .animate-blink {
    animation: none;
  }
}
```

Et conditionner les animations SVG en JS sur la valeur de `window.matchMedia('(prefers-reduced-motion: reduce)')`.

## 6. Cadre légal français

**Loi pour une République numérique (article 47)** + décret 2019-768 :

- **Obligation RGAA AA** pour : services publics, entreprises de plus de 250 M€ de CA, délégataires de service public.
- **Pour un acteur privé < 250 M€ CA** (cas roxanetnous) : **pas d'obligation légale formelle**, mais :
  - Future directive européenne **Acte sur l'accessibilité (EAA)** entrée en vigueur **28 juin 2025** : couvre les services e-commerce et bancaires ; **roxanetnous (marketplace de service) est dans le périmètre potentiel**.
  - Sanction CNIL possible sur le terrain de la non-discrimination si exclusion d'usagers handicapés démontrable.

**Conclusion juridique** : pas d'obligation immédiate prouvée, mais **risque réel** sur l'EAA et **enjeu réputationnel/éthique** clair pour une marketplace de services à la personne (population âgée et handicapée parmi les bénéficiaires).

## 7. Effort de remédiation estimé

Effort exprimé en jours-développeur, base codebase actuel.

### Lot A — « Quick wins » (3-4 j)
- Skip-link + `<main>` cohérent au layout
- Couleur de focus globale conforme (3:1 minimum)
- `prefers-reduced-motion` dans globals.css

### Lot B — « Conformité fonctionnelle » (5-7 j)
- Onboarding multi-étapes : `role="progressbar"`, gestion focus inter-étapes, `role="alert"` erreurs
- Messagerie : `aria-live`, `role="log"`, annonces nouveaux messages
- Tous les formulaires : audit champ par champ, association labels
- Cookie banner : `role="region"` + amélioration UX
- Description textuelle alternative pour la carte hero
- Modales et toasts du dashboard : rôles ARIA

### Lot C — « Excellence » (3-5 j)
- Tests automatisés axe-core dans Playwright
- Page de déclaration accessibilité (article 47)
- Tests manuels VoiceOver / NVDA documentés (1 parcours auxiliaire + 1 parcours bénéficiaire)
- Mode contraste élevé optionnel
- Audit Leaflet et alternatives clavier sur la recherche

**Total** : **11-16 jours-développeur** pour atteindre WCAG 2.2 AA réelle sur les parcours critiques.

## 8. Recommandation

### Reco principale : **Option B amendée — NFR transverse + Lot A immédiat**

1. **Maintenant (avant épic 3)** : exécuter le **Lot A « Quick wins »** comme micro-épic d'1 sprint (3-4 j). Bénéfice : remet la base sémantique à niveau avant que de nouveaux écrans n'aggravent la dette.
2. **Inscrire au PRD** un NFR « WCAG 2.2 AA sur les parcours critiques » via `bmad-testarch-nfr` et `bmad-edit-prd`.
3. **Chaque story future** inclut une definition-of-done a11y : champs labellisés, focus géré, erreurs annoncées, contrastes vérifiés.
4. **Lot B** : étalé sur l'épic 3, au fur et à mesure que les zones concernées sont touchées (messagerie, dashboards).
5. **Lot C** : à l'horizon épic 4 ou 5, quand le produit a stabilisé.

### Pourquoi pas un épic dédié ? (Option A rejetée)
- Retarderait de 11-16 jours le delivery business.
- Risque de bâclage si fait en bloc sous pression.
- L'a11y se dégrade naturellement si elle n'est pas dans la culture du flux normal — un épic one-shot ne résout pas ce problème de fond.

### Pourquoi pas l'Option B pure ? (rejetée)
- Le code existant est tellement en dessous du minimum sémantique qu'attendre que chaque story le corrige « au passage » ne couvrira jamais les pages déjà livrées (landing, messagerie, onboarding).
